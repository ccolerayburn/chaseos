  - --script
  - --smoke wallpapers
  - --smoke startup
- Do not require admin rights.
- Prefer a lock file or named mutex approach that works on Windows.
- Handle stale locks gracefully.

Suggested path for a lock file if using file lock:

%LOCALAPPDATA%\ChaseOS\runtime\chaseos.lock

If using a Windows named mutex, use a ChaseOS-specific name.

Add tests for stale lock handling where practical.

Step 4: Add safe Windows startup shortcut management

Add commands:

/startup status
/startup enable
/startup disable

These commands should manage a per-user Windows Startup folder shortcut only.

Startup folder target:

%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup

Shortcut name:

ChaseOS.lnk

Behavior:
- /startup status reports whether the shortcut exists and what it points to.
- /startup enable creates or updates the shortcut to launch ChaseOS normally.
- /startup disable removes the shortcut if present.
- No admin rights required.
- Do not modify registry Run keys.
- Do not create scheduled tasks.
- Do not modify system-wide startup.
- Do not touch taskbar pins/icons.

Shortcut target:
- Prefer the active venv pythonw.exe or python.exe if the project is still source-run based.
- Use pythonw.exe if available to avoid a console window.
- Arguments should launch:
  -m chaseos

Working directory should be:
C:\_Codex\chaseos

If creating .lnk files requires a dependency, prefer using Windows COM via pywin32 only if already available. If not available, use a minimal PowerShell or COM approach without adding heavy dependencies. Keep it robust and tested with abstraction/mocking.

The command should clearly report:
- shortcut created/updated/removed
- target path
- arguments
- working directory
- no registry changes
